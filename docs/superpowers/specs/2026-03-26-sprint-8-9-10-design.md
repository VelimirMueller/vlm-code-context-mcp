# Sprint 8-9-10 Design Spec: Discovery → React Rewrite → Eye Candy

**Date:** 2026-03-26
**Milestone:** 3 (Ecosystem Growth — Dashboard Modernization)
**Velocity target:** 19pt per sprint
**Total scope:** 57pt across 3 sprints, 15 tickets

## Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Sprint order | Discovery → React → Eye Candy | Discovery maps scope, React is clean rewrite, animations are final polish — nothing gets rewritten twice |
| React setup | Vite + React + TypeScript + Zustand | Full standalone app with own build pipeline; MCP server serves the built bundle |
| Animation library | Framer Motion (~30KB) | Spring physics, AnimatePresence, layout animations. Sweet spot between polish and bloat |
| Discovery scope | Audit + build missing MCP tools | No throwaway frontend prototypes; React sprint gets a complete API surface |
| File explorer default | All collapsed, auto-expand on search/select | Progressive disclosure with smart reveal |

## Retro Lessons Applied

These rules from 665 retro findings across 10 sprints are mandatory for all 3 sprints:

- **19pt velocity cap** — no over-commitment (S2 burned at 27pt)
- **8pt max per individual dev** — prevents capacity imbalance
- **No subagent file writes** — all src/ changes written directly, verified with build
- **Retro enforced at sprint close** — sprint cannot close without retro findings
- **Query ticket IDs before updating** — never assume sequential IDs
- **Build verification after every change** — `npm run build` + `npm test`
- **Parallel QA agents** — 4 simultaneous for verification speed
- **Checkpoint after first 2 tickets** — verify output on disk before continuing

### T-053: React Component Tree Design (3pt)
**Assignee:** Frontend Developer
**Acceptance criteria:**
- COMPONENT_TREE.md committed to `.claude/scrum/` with:
  - Atomic design hierarchy: atoms → molecules → organisms → templates → pages
  - Every component named, typed, with props interface sketched
  - Smart vs dumb component classification for each
  - File/folder structure following:
    ```
    src/dashboard/app/
    ├── components/
    │   ├── atoms/        # Button, Badge, Stat, Dot, Icon
    │   ├── molecules/    # StatGroup, FileItem, TicketCard, AgentCard
    │   ├── organisms/    # FileTree, KanbanBoard, BentoGrid, TeamGrid
    │   └── templates/    # ExplorerLayout, SprintLayout, PlanningLayout
    ├── pages/            # CodeExplorer, Sprint, ProjectManagement
    ├── hooks/            # useFiles, useSprints, useAgents, useSearch
    ├── stores/           # fileStore, sprintStore, uiStore
    ├── lib/              # api.ts, utils.ts, constants.ts
    └── types/            # shared TypeScript interfaces
    ```
  - Which vanilla JS functions map to which React components
  - Extraction plan: what logic becomes hooks, what becomes utils

## Sprint 9 — React Rewrite (19pt)

**Goal:** Replace `dashboard.html` with a full React + TypeScript application. Every existing feature preserved, interactive workflows added.

**Prerequisites:** Sprint 8 deliverables (AUDIT.md, COMPONENT_TREE.md, STORE_DESIGN.md, new MCP tools)

### T-056: Vite + React + TS Scaffold (5pt)
**Assignee:** Architect + Frontend Developer
**Acceptance criteria:**
- `src/dashboard/app/` directory with Vite config
- `package.json` updated: new scripts (`dashboard:dev`, `dashboard:build`)
- Vite dev server proxies `/api/*` and `/api/events` (SSE) to MCP dashboard server
- Build output goes to `dist/dashboard/` (replaces old `dashboard.html`)
- Dashboard server updated to serve Vite build output (index.html + assets)
- TypeScript strict mode, path aliases (`@/components`, `@/hooks`, etc.)
- Tailwind CSS configured with existing design tokens (--bg, --surface, --accent, etc.)
- ESLint + Prettier configured
- `npm run build` compiles both MCP server and dashboard app

### T-057: Zustand Stores + API Hooks (5pt)
**Assignee:** Backend Developer + Frontend Developer
**Acceptance criteria:**
- All stores from STORE_DESIGN.md implemented:
  - `fileStore` with actions: `fetchFiles`, `selectFile`, `fetchFileDetail`, `fetchChanges`
  - `sprintStore` with actions: `fetchSprints`, `selectSprint`, `fetchTickets`, `fetchRetro`
  - `agentStore` with actions: `fetchAgents`
  - `planningStore` with actions: `fetchMilestones`, `fetchVision`, `fetchGantt`, `updateVision`, `createMilestone`, `linkTicket`
  - `uiStore` with actions: `setPage`, `setTab`, `toggleSidebar`, `setSearch`, `toggleFolder`
- Custom hooks: `useFiles()`, `useSprints()`, `useAgents()`, `usePlanning()`, `useSearch()`
- SSE hook: `useEventSource()` that updates relevant stores on `'updated'` events
- URL hash sync: `useHashRouter()` that reads/writes page + tab + file ID
- API client: `lib/api.ts` with typed fetch wrapper, error handling, request deduplication
- All hooks have proper TypeScript return types

## Sprint 10 — Eye Candy & Documentation (19pt)

**Goal:** Add enterprise-grade transitions, animated hero text per tab, polished micro-interactions, and update all documentation.

**Prerequisites:** Sprint 9 complete (full React app functional)

### T-061: Page Transitions with Framer Motion (5pt)
**Assignee:** Frontend Developer
**Acceptance criteria:**
- `framer-motion` added as dependency (Manager pre-approved)
- `AnimatePresence` wraps page router — pages fade/slide on switch
- Sub-tab transitions: content cross-fades with spring physics
- Sidebar collapse/expand animated
- Kanban card drag feels physical (spring overshoot)
- No animation on reduced-motion preference (`prefers-reduced-motion`)
- Page transitions complete in < 300ms (perceived instant)

### T-062: Tab Hero Animations (5pt)
**Assignee:** Frontend Developer
**Acceptance criteria:**
- Each page/tab shows an animated hero text block on mount:
  - **Code Explorer:** "Tracking {N} files across {M} directories — {X} exports indexed" (count-up animation)
  - **Sprint Board:** "Sprint {name} — {done}/{total} tickets shipped, {velocity}pt velocity" with staggered word reveal
  - **Retro Insights:** "665 findings across 10 sprints — here's what we learned" with typewriter effect
  - **Project Management:** "Milestone {current} — {progress}% complete, {days} days remaining" with progress ring animation
  - **Team:** "{N} agents, {mood} average mood — {active} active now" with agent icons floating in
- Text is dynamic, pulled from store data
- Animations are spring-based (Framer Motion), staggered children
- Fits existing dark theme, uses Geist Sans/Mono
- Hero area is compact (max 80px height) — informative, not decorative

### T-063: Micro-Interactions (3pt)
**Assignee:** Frontend Developer
**Acceptance criteria:**
- Card hover: subtle lift (translateY -2px) + shadow increase, spring easing
- Stat numbers: count-up animation on mount (0 → actual value over 400ms)
- Skeleton loaders: shimmer effect replacing static pulse
- Button press: scale(0.97) on active
- Toast notifications: slide-in from bottom-right for save confirmations
- Tab indicator: animated underline slides between tabs (not instant switch)
- File tree expand/collapse: height animation with spring
- All animations respect `prefers-reduced-motion`

### T-064: Landing Page Animation Rewrite (3pt)
**Assignee:** Frontend Developer
**Acceptance criteria:**
- Replace CSS `@keyframes` landing animation with Framer Motion orchestrated sequence
- Same 5 phases but with spring physics and smoother timing:
  1. Grid lines draw in (stagger)
  2. Code file nodes fade up with scale spring
  3. Sprint board slides in from right
  4. Agent circles pop in with overshoot
  5. Feature labels typewriter in
- Total animation duration: 3-4 seconds
- Skip animation button (or click-to-skip)
- Animation plays once per session (stored in sessionStorage)

## Technical Constraints

- **Dependencies require Manager approval** — Vite, React, Zustand, Framer Motion are pre-approved for these sprints. Any additional deps need review.
- **Build must stay under 10s** — Vite builds are typically < 3s; monitor this.
- **Bundle size budget** — React (~40KB) + Zustand (~2KB) + Framer Motion (~30KB) = ~72KB gzipped. Acceptable for a developer tool.
- **No breaking API changes** — dashboard server endpoints stay identical. React app is a frontend-only change.
- **SSE must work** — Vite dev proxy must correctly handle EventSource connections.

## Process Rules

Each sprint follows the 5-day cycle from SPRINT_PROCESS.md:
1. Day 1: Planning (PO presents, team estimates, self-assign, commit to 19pt)
2. Day 2-3: Implementation (autonomous, 30-min escalation, checkpoint after 2 tickets)
3. Day 4: QA + Polish (parallel QA agents, bug triage, security review)
4. Day 5: Ship + Retro (sign-off, summary, retrospective with actionable findings)

Sprint cannot close without:
- All tickets DONE or explicitly deferred with reason
- QA verification on each ticket
- Retro findings committed via `add_retro_finding` MCP tool
- Build passing (`npm run build` + `npm test`)
